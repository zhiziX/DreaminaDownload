# 智子X 即梦解析 - 四类资源解析策略

## 1. 当前实现的总原则

当前实现已经明确区分 4 类资源：

| 类别 | 站点 | 媒体类型 | 识别入口 |
| --- | --- | --- | --- |
| 即梦主站图片 | jimeng.jianying.com | image | `/ai-tool/work-detail/<id>` + `workDetailType=image` 或 `itemType=9/109` |
| 即梦主站视频 | jimeng.jianying.com | video | `/ai-tool/work-detail/<id>` + `workDetailType` 含 `video` 或 `itemType=53/210` |
| Dreamina 图片 | dreamina.capcut.com | image | `/ai-tool/image/<id>` 或 `/ai-tool/work-detail/<id>` + 图片参数 |
| Dreamina 视频 | dreamina.capcut.com | video | `/ai-tool/video/<id>` 或 `/ai-tool/work-detail/<id>` + 视频参数 |

页面识别入口在 `content.js` 的 `getPageStrategy()`。

当前模式不是“全站抓取”，而是“只盯当前详情 + 只返回一个主媒体”。

## 2. 当前工具到底是如何查询的

### 2.1 被动触发部分

被动触发负责发现“现在该重新查了”。当前即梦已经不只是路由监听，还包含主世界里的详情桥接。

| 触发器 | 文件 | 作用 |
| --- | --- | --- |
| SPA 路由变化监听 | `page-bridge.js` | 劫持 `history.pushState`、`history.replaceState`，监听 `popstate`、`hashchange`，向 content script 发 `jimeng-page-change` |
| 主世界详情 bridge | `page-bridge.js` | 在页面主世界读取当前详情 workDetail，并通过 `window.postMessage` 发 `jimeng-work-detail` |
| 主世界 `_ROUTER_DATA` 赋值钩子 | `page-bridge.js` | 监听运行时 `window._ROUTER_DATA` 更新，第一时间桥接当前详情结构化数据 |
| `mget_item_info` 响应拦截 | `page-bridge.js` | 拦截 fetch / XHR，命中即梦 `mget_item_info` 时直接桥接当前详情结构化数据 |
| DOM 变化监听 | `content.js` | `MutationObserver` 监听 `childList/subtree/attributes`，属性包含 `src`、`poster`、`style`、`class`、`data-src`、`data-original` |
| popup 主动刷新 | `popup.js -> content.js` | 用户点刷新后，content script 重新拉当前详情数据 |

结论：

- 有被动监听
- 即梦现在有被动拦截 fetch/XHR 响应，但只用于 `mget_item_info` 当前详情桥接
- 没有 hidden tab 模拟刷新
- 没有第三方图片补源接口

### 2.2 主动查询部分

当前真正拿下载链接，依赖这些路径：

| 路径 | 是否主动 | 适用范围 | 说明 |
| --- | --- | --- | --- |
| 当前详情层主媒体 DOM 锁 | 是 | 即梦图片、即梦视频 | 先锁定“当前正在看的详情是谁”，用于过滤旧详情和列表残留媒体 |
| 当前文档内联 `_ROUTER_DATA` | 是 | 即梦图片、即梦视频、Dreamina 图片、Dreamina 视频 | 即梦当前主链路，直接从当前文档 script / HTML 里的 payload 提最佳链接 |
| 读取页面内现成 `window._ROUTER_DATA` | 是 | 即梦图片、即梦视频、Dreamina 图片、Dreamina 视频 | 当前页面运行时内存里的结构化详情 |
| `fetch(location.href, { cache: 'no-store' })` 重拉当前详情 HTML，再抽 `_ROUTER_DATA` | 是 | 即梦图片、即梦视频、Dreamina 图片、Dreamina 视频 | 当前页面还没把详情数据放进内存时，用当前详情页 HTML 补一遍 |
| 主世界 bridged workDetail | 半主动 | 即梦图片、即梦视频 | 来自主世界 router 或 `mget_item_info`，现在只作低优先级补源和调试线索 |
| Dreamina 官方 `get_item_info` | 是 | Dreamina 图片、Dreamina 视频 | Dreamina 的官方 detail 接口 |
| 直接读取当前详情层主媒体 DOM | 是 | 全部四类 | 最后回退，图片读 `currentSrc/src/srcset/data-*`，视频读 `currentSrc/src/source` |

结论：

- Dreamina：官方接口 + 当前文档/运行时 router + HTML + DOM
- 即梦：DOM-first 当前详情锁 + 当前文档内联 `_ROUTER_DATA` + `window._ROUTER_DATA` + HTML + bridge + DOM
- 即梦当前主目标已从“稳定打出旧 `router-*` 标签”切到“稳定拿到当前详情最佳链接”，`source` 只保留 provenance 语义
- 当前最危险的问题不是“拿不到数据”，而是没有稳定锁住“当前详情是谁”就让旧结构化数据提前短路

## 3. 下载链接是如何获得的

### 3.1 统一流程

统一流程如下：

1. 先用 `getPageStrategy()` 判断属于哪一类详情页。
2. 即梦先用 `getMainImageElements()` / `getMainVideoElements()` 锁定当前详情主媒体，并生成详情锁 signatures。
3. 用 `getPublishedItemId()` 或详情锁派生当前详情 identity。
4. 如果发现详情切换，清空旧缓存，重新 hydrate。
5. Jimeng 从 `当前文档内联 _ROUTER_DATA -> window._ROUTER_DATA -> HTML -> bridge` 收集全部命中当前详情的结构化 workDetail。
6. `scanMedia()` 根据图片或视频策略，直接从这些结构化 workDetail 里比较最佳候选。
7. 所有结构化候选都失败后才退回 DOM。
8. 最终返回 `mediaItems[0]`。

### 3.2 当前详情主媒体如何定位

当前不是抓整页所有图片/视频，而是只抓详情层主媒体。

| 步骤 | 规则 |
| --- | --- |
| 列表残留过滤 | 即梦先排除 `masonry/feed/list/card/gallery/recommend` 等背景媒体容器 |
| 详情层过滤 | 只优先保留位于 `dialog/modal/drawer/detail` 容器或 fixed/sticky 覆盖层中的媒体元素 |
| 主图粗筛 | 图片宽高至少约 `220 x 220`，且可见 |
| 主视频粗筛 | 视频宽高至少约 `260 x 160`，且可见 |
| 同类候选排序 | 详情容器优先，其次 fixed 覆盖层优先；即梦再补一层视口中心距离优先，最后按显示面积最大排序 |
| 最终结果 | 每类只保留 1 个主元素 |

## 4. 四类资源的获取方法和降级层级

## 4.1 Dreamina 图片

### 获取方法

Dreamina 图片共有 4 条获取路径：

1. Dreamina 官方 `get_item_info`
2. 当前页内 `_ROUTER_DATA`
3. 重拉当前详情 HTML 后抽 `_ROUTER_DATA`
4. 当前详情层主图 DOM

### 固定降级链

1. `largeImages`
2. `coverUrlMap`
3. `coverUrl`
4. DOM 主图

### 当前实现特征

- Dreamina 图片是四类里唯一明确带 `official-*` 标签的类别
- 会做 assetKey 匹配，尽量保证拿到的是当前这张图，不是上一张图
- 如果官方数据已命中，Router 侧匹配会放宽；如果官方没命中，Router 侧匹配更严格

## 4.2 即梦主站图片

### 获取方法

即梦主站图片当前共有 5 条获取路径：

1. 当前文档内联 `_ROUTER_DATA`
2. 当前页运行时 `window._ROUTER_DATA`
3. 重拉当前详情 HTML 后抽 `_ROUTER_DATA`
4. 主世界 bridge 当前详情 workDetail
5. 当前详情层主图 DOM

### 固定降级链

1. `largeImages`
2. `coverUrlMap`
3. `coverUrl`
4. DOM 主图

DOM 回退顺序：

1. `img-currentSrc`
2. `img-srcset`
3. `img-src`
4. `img-data-*`

### 当前 source 标签

| 层级 | source 标签 |
| --- | --- |
| 当前文档结构化大图 | `router-largeImages` |
| 运行时 router 大图 | `router-runtime-largeImages` |
| HTML 补源大图 | `html-router-largeImages` |
| bridge / API 大图 | `bridge-api-largeImages` 这类 |
| 当前文档结构化尺寸图 | `router-coverUrlMap-4096` 这类 |
| 运行时 router 尺寸图 | `router-runtime-coverUrlMap-4096` 这类 |
| HTML 补源尺寸图 | `html-router-coverUrlMap-4096` 这类 |
| bridge / API 尺寸图 | `bridge-api-coverUrlMap-4096` 这类 |
| 当前文档结构化封面图 | `router-coverUrl` |
| 运行时 router 封面图 | `router-runtime-coverUrl` |
| HTML 补源封面图 | `html-router-coverUrl` |
| bridge / API 封面图 | `bridge-api-coverUrl` 这类 |
| DOM 当前图 | `img-currentSrc` |
| DOM srcset | `img-srcset` |
| DOM src | `img-src` / `img-attr` |
| DOM data 属性 | `img-data-src` / `img-data-original` / `img-data-full` |

### 当前实现特征

- 即梦主站图片现在先用 DOM 锁定当前详情，再从所有命中当前详情的结构化 workDetail 里直接挑最佳图片链接
- `mget_item_info` 里的 `cover_url_map` / `cover_url` 仍是即梦 SPA 当前详情的重要结构化来源，但现在只作低优先级补源
- 当前详情页 SSR / 内联脚本常在 CSR 入口真正执行前就已带上正确 workDetail
- strict assetKey 不再反向控制即梦结构化图片选源；source 只用于 provenance 展示

## 5. 来源优先级、失效条件、短路条件

| 来源 | 站点 | 优先级角色 | 何时可用 | 何时失效 |
| --- | --- | --- | --- | --- |
| Dreamina 官方 `get_item_info` | Dreamina | 最高 | itemId 命中、接口成功 | itemId 变化 |
| 当前文档内联 `_ROUTER_DATA` | 全部 | 当前页第一结构化候选 | 当前详情的 SSR / 内联脚本已写入正确 workDetail | 当前文档还没切到新详情，或内联数据仍是旧详情 |
| `window._ROUTER_DATA` | 全部 | 次级结构化来源 | 当前页运行时内存里已有当前详情数据 | 页面运行时内存尚未更新到当前详情 |
| HTML 提取 `_ROUTER_DATA` | 全部 | 补源 | 当前详情页 HTML 已包含当前详情结构化数据 | 抓到的仍是旧详情 |
| bridged workDetail | 即梦 | 低优先级补源 | `publishedItemId` 或 `detailMediaKey` 与当前详情匹配，且未过期 | 路由变化、主媒体 key 变化、TTL 过期、身份不匹配 |
| DOM 主媒体 | 全部 | 最后回退 | 结构化来源都不匹配当前详情 | 不作为当前详情身份的首选依据 |

## 6. 当前最重要的故障模型

这次即梦 SPA 问题的根因模型已经冻结为：

1. `page-bridge.js` 在新 URL 下可能拿到旧 detail
2. 旧 detail 被带着新 URL 发进 content script
3. `content.js` 如果只按 URL 信任 bridge，就会把旧 detail 写成当前详情缓存
4. `refreshActiveDetailData()` 被旧 bridge 提前短路
5. 当前详情拿不到匹配结构化数据
6. 当前详情拿不到匹配结构化数据，或者虽已在当前文档 SSR / 内联脚本里出现却没被及时采用
7. 最终退回 `img-currentSrc` / `video-currentSrc`

所以这次修复的关键不是继续堆新来源，而是：

- 先用 DOM 锁定当前详情主媒体
- 再收集当前文档、运行时 router、HTML、bridge 里所有命中当前详情的结构化 workDetail
- 直接比较这些结构化 workDetail 里的最佳正式链接
- `source` 只保留 provenance 语义，不再反向控制选源
- bridge 退成低优先级补源
- 最后才退回 DOM

## 7. 最短结论

当前实现不是简单的“router/html + DOM”。

- Dreamina：官方接口 + 当前文档/运行时 router + HTML + DOM
- 即梦：DOM-first 当前详情锁 + 当前文档内联 `_ROUTER_DATA` + `window._ROUTER_DATA` + HTML + bridge + DOM

当前最核心的稳定性原则是：

**先锁定当前详情，再直接比较命中当前详情的结构化最佳链接；`source` 只作 provenance，结构化来源优先于 DOM 回退。**
